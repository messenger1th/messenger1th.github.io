TCP握手为什么是三次

原因一：确保双方能够发送和接受syn和ack数据包，并同步seq数据。

在网络中，由于丢包的存在，发出一个数据包，在没有收到应答的情况下，无法确定对方是否收到。

因此，要确认一个数据包对方收到了，对于发送方来说，只有收到接收方的ACK确认包。

请求建立连接

  • 发送方发送syn包用于发起连接。
  • 接收方发送ack包用于同意连接建立。

即，发送和确认，一来一回。保证了一个数据包被接受。

对于主动发起方和被动发起方来说，都需要一来一回的过程。本来是四次握手，但由于第二次握手时的ack和（原来是）第三次握手的syn合并成了一次，总共也就是一次握手。

原因二：避免历史连接

RFC 793: The principle reason for the three-way handshake is to prevent old duplicate connection initiations from causing confusion.

假设一个syn在发送时延时，发送方认为接收方没有接收到该包。即认为该包失效，重发syn包。

但该包在新syn包之前到达，如果是两次握手，就会建立历史连接。

三次握手就可以避免，流程如下

  1. 旧syn到达，被动发起方发送ack+syn。
  2. 主动发起方接收到该ack+syn，认识到该连接过期（序列号过期或超时），便发送RST给被动发起方。
  3. 被动发起方接到RST，退出SYN_RCVD状态，避免了历史连接。

原因三：避免资源浪费

如果只有两次握手，即每收到一个 SYN 就只能先主动建立一个连接，这会造成什么情况呢？

消息滞留情况下，服务器重复接受无用的连接请求 SYN 报文，而造成重复分配资源。

其他原因

对于两次握手，在面对SYN攻击时，即建立多个无用TCP连接，也会造成资源大量浪费，导致服务器崩溃无法为正常用户服务。

TCP挥手为什么是四次？

类比TCP三次握手，对于TCP连接双方，都需要一个FIN和一个ACK。

FIN发送方，表示自己不在发送数据但可以接收数据。

通常来说，双方的数据不是同时发送完成的，因此FIN和ACK一般是分开发送。

特定情况下（TCP延迟应答开启+无数据发送或第二次FIN到达），便可缩短成三次握手。见4.22 TCP 四次挥手，可以变成三次吗？ | 小林coding (xiaolincoding.com)

TCP和UDP的区别

我觉得TCP和UDP最大的区别就是，应用场景。

对于交易支付这种依赖可靠性的传输，需要用到TCP。对于视频这种对于延迟有较低要求的场景，需要用到UDP。

针对TCP的可靠，UDP的低延迟，展开对区别的描述。

TCP可靠性

可靠意味着对于文件要完整地无差错到达。

  1. 而数据链路层是有数据包大小限制的。对于一个大文件，我们需要将其分片，编号。保证完整到达只需编号完整即可。

  2. 数据分片传输，也就导致了TCP是流式协议。

  3. 需要对以及接受到的数据包编号进行维护，TCP是面向连接的协议也就不难理解了。

  4. 对于丢失的数据包，需要进行重传。而UDP不会。

UDP低延迟

低延迟，只管立即发送，不管能不能完整到达，不管网络拥塞情况。

  • 对于TCP的那套建立连接的三次握手，就可以省去，所以UDP不是面向连接的协议。
  • 是“自私”的协议，没有拥塞控制、流量控制。在网络拥挤时，仍然不减少发送速率。
  • 没有了编号，数据包就可能是乱序的。
  • 没有了重传，不管丢包问题。

当然还有因为功能的不一致，导致的区别。

  1. 首部开销：TCP需要维护Ack和Seq，关闭连接的FIN和RST等大概20字节，如果用到”选项“，则会更多。而UDP首部（共8字节）只需要双方端口、包长度、检验和各16位。
  2. 分片不同：TCP在传输层分片，而UDP在网络层分片。由于网络层不保证可靠性，如果TCP在网络层再分片，原本为一个编号的数据分为多个包，若丢失一个就要全部重传，因而TCP在传输层就直接分片进行编号，丢失只需重传该编号的即可。而UDP本就不保证可靠，就在网络层分片即可。

按照各种特性分，TCP和UDP的总区别如下

1. 连接

  • TCP 是面向连接的传输层协议，传输数据前先要建立连接。
  • UDP 是不需要连接，即刻传输数据。

2. 服务对象

  • TCP 是一对一的两点服务，即一条连接只有两个端点。
  • UDP 支持一对一、一对多、多对多的交互通信

3. 可靠性

  • TCP 是可靠交付数据的，数据可以无差错、不丢失、不重复、按序到达。
  • UDP 是尽最大努力交付，不保证可靠交付数据。

4. 拥塞控制、流量控制

  • TCP 有拥塞控制和流量控制机制，保证数据传输的安全性。
  • UDP 则没有，即使网络非常拥堵了，也不会影响 UDP 的发送速率。

5. 首部开销

  • TCP 首部长度较长，会有一定的开销，首部在没有使用「选项」字段时是 20 个字节，如果使用了「选项」字段则会变长的。
  • UDP 首部只有 8 个字节，并且是固定不变的，开销较小。

6. 传输方式

  • TCP 是流式传输，没有边界，但保证顺序和可靠。
  • UDP 是一个包一个包的发送，是有边界的，但可能会丢包和乱序。

7. 分片不同

  • TCP 的数据大小如果大于 MSS 大小，则会在传输层进行分片，目标主机收到后，也同样在传输层组装 TCP 数据包，如果中途丢失了一个分片，只需要传输丢失的这个分片。
  • UDP 的数据大小如果大于 MTU 大小，则会在 IP 层进行分片，目标主机收到后，在 IP 层组装完数据，接着再传给传输层。

如何应对Syn泛洪攻击？

Syn泛洪攻击，是大量伪造的IP和端口，向服务器发送Syn报文。服务器便会将他们放入半连接队列，占满服务器的半连接队列后，就无法为正常TCP连接服务。

因此，针对上述过程，有以下方式应对。

既然是占满了服务器的半连接队列

  • 增大半连接队列的大小，Linux下，由内核的多个参数决定。
  • 服务器的空间满了，那就调大网卡的队列大小。Linux下，由参数 net.core.netdev_max_backlog决定。

既然空间不够用，那就时间换空间

  • 开启 net.ipv4.tcp_syncookies。在半连接队列满了之后，服务不会丢弃syn报文，而是根据算法计算一个cookie值。在下次收到ack报文时，验证合法性。

当然还有一些可能误伤的方法

  • 减少syn+ack报文的重传次数。由 tcp_synack_retries内核参数决定（默认值是 5 次）
  • 减少重传时间等等。

安全相关算法

对称加密算法

有对每固定数目的数据块加密的算法，称为分组秘密（或块密码），也有对流进行加密的算法，称为为流密码。

  • DES（逐渐被AES取代）：是块密码
  • AES，块密码，针对块大小有AES-128、AES-192、AES-256等。

非对称加密算法

  • RSA
  • DH->DHE->ECDHE

哈希算法

  • MD5
  • SHA-1、SHA-2、SHA-3。

HTTPS如何保证安全？

HTTP不安全的方面有3个

  • 可能会被窃听
  • 身份可被冒充
  • 数据可被篡改

对于一次通信，我们首先保证双方不被窃听。

使用对称加密直接传输密钥会有密钥也被窃取的风险。因此改用非对称加密，私钥 + 公钥来加密。

但由于非对称加密直接加密内容运算量太大，我们改用非对称加密加密对称密钥的方式，该对称密钥称为，对话密钥。

这样，就解决了通信会被窃听的风险。但仍然存在被冒充的风险。

数字证书CA，解决被冒充的风险。

非对称加密，仅仅能让通信不被窃听。但伪造一对公钥私钥，冒充服务器还是可行的。需要对这进行安全措施。

解决方案是，CA提供服务，对服务器公钥进行验证。

服务器注册步骤如下：

  1. 服务器向CA发送其公钥、用途、颁发者、有效时间等信息，称为证书信息。
  2. CA用Hash算法做一个摘要（也称签名）。
  3. 同时，使用CA的私钥，对该摘要进行加密。

客户端验证步骤如下

  1. 接收到服务器发来的数字证书。
  2. 使用CA的公钥（通常由浏览器或操作系统集成），将数字证书里头的摘要解密，得到CA注册时的原始Hash摘要值。
  3. 使用Hash算法同样对原始证书进行摘要计算。
  4. 比较两个摘要是否相同。

这样就保证了，通讯过程的安全性和服务器的权威性，但还可能被篡改。

哈希算法保证完整性

在通讯双方安全地生成会话密钥后，就可以对通信进行加密了。

每次对数据分块进行压缩，以及该块的Hash值进行计算。再用会话密钥进行加密。

HTTPS握手流程

以RSA算法为例，但实际情况下，使用ECDHE算法更普遍。

抓包分析：

TCP三次握手

SSL/TLS握手

  1. Client Hello ， 发送以下数据
    • 支持的SSL/TLS协议版本
    • 随机数Client Random
    • 支持的密码套件列表，如RSA。
  2. Server Hello，发送以下数据
    • 确定的SSL/TLS协议版本
    • 随机数Server Random
    • 确定的密码套件，如RSA。
    • 服务器证书（含公钥等信息）。
  3. 第三次握手：客户端回应
    • 客户端检查数字证书的有效性，并获取公钥。
    • Change Cipher Key Exchange：生成一个随机数pre-master，并用公钥加密发。仅仅能用私钥打开，所以保证安全。
    • Change Cipher Spec：告诉服务端开始使用加密方式发送消息。
    • Encrypted Handshake Message（Finished）把之前所有发送的数据做个摘要，再用会话密钥（master secret）加密一下，让服务器做个验证。
  4. 第四次握手：服务器回应
    • Change Cipher Spec：告诉客户端开始使用加密方式发送消息。
    • Encrypted Handshake Message（Finished）把之前所有发送的数据做个摘要，再用会话密钥（master secret）加密一下，让客户端做个验证。

注意到，以上通信安全的前提是，服务器私钥不被泄露。一旦私钥泄露，之前所有的通信都会被破解。

于是，DH（演变: DH->DHE->ECDHE）算法出现了。它的通信双方都使用公私钥。服务器的公私钥不变，仍然用于确认身份。但客户端的公私钥动态选取。

这样，就算泄露了服务器私钥。以前的通信，还需要客户端的私钥才能破解，保证了前向安全性。

HTTP发展史

HTTP1.0

是一份参考文档，并未强制标准。定义了HTTP的部分规范。

直接由明文传输，不安全，无状态信息，一个请求需要TCP握手挥手，效率很低。

HTTP/1.1

对HTTP/1.0进行了连接的优化，有了HTTP长连接, mo’ren，多个请求可用同一个TCP连接。HTTP/1.0默认不开启，需要在头部字段开启。

同时，理论上支持管道传输，无需上一个请求的回应即可进行下一个请求，但实际上默认不开启，浏览器基本不支持。

HTTP2

HTTP2基于HTTPS，默认开启SSL/TLS。安全性有了保障。

HTTP2使用HPACK算法，通信双方都有一张固定静态表、维护一张动态表、使用Huffman编码对头部压缩优化。

HTTP2使用二进制帧，将原来的文本改为二进制流，便于直接传输，无需中间转换。

HTTP2可以实现并行传输，给原本HTTP数据压缩后，给封装成二进制帧frame，含帧长度、流标识符、帧类型、标志位等信息，一条TCP连接可以传输多个stream。

此外，HTTP2还支持服务器主动推送资源。减少了一次请求。

路由算法

已知，网络是划分区域的，划分成一个一个的AS（Autonomous System）自治系统。

一般来说，AS与AS之间使用BGP（是一种EGP协议）算法路由，AS内使用RIP或OSPF算法(统称IGP协议)路由。

RIP算法

Routing Information Protocol：基于距离，泛洪法广播自己的路由表信息。

  • 仅和相邻路由器交换信息。
  • 交换的信息是当前本路由器所知道的全部信息，即自己现在路由表。
  • 按固定的时间间隔交换信息，如每隔30s或网络拓扑发生变化时。

最终路由器都具有其他路由器距离信息。

特点：好消息传得快，坏消息传得慢

由于路由表中的信息并不记录具体的路线，只关心到达的距离。

在新加路线时，路由表会被泛洪很快更新为更优路径。

但线路故障时，虽然故障双方能检测到故障，设置对方不可达。

但其他路由器并不知道，在自己的广播周期时，发送以前的老路由表。现在实际不可达的路线被认为是可达，就发生了错误。

改良方式

  1. 水平分割：从上一个节点来的信息，更新节点后，不再发送更新的节点回去。
  2. 毒性反转的水平分割：接受节点信息时还是会回传，但回传一个无穷大的值。

但水平分割的方式在有环路时还是存在问题。故RIP算法现在并不流行。

OSPF算法

Open Shortest Path First：基于链路状态，首先确定自己的链路状态，再泛洪广播LSA信息。

最终，每个路由器都有所有节点的邻接表信息，这样就可以算出最短路径，比如用Dijkstra算法。

故障：当链路故障时，只需更新自身的邻接表，然后泛洪发送给其他节点更新即可。

新增：同理。
